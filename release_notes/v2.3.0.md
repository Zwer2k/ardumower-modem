# Release Notes v2.3.0

## Map Editor: Dirty State, Transient Maps, and Deletion

### New / Improved

- **Transient Maps (RAM-only) for intercepted maps**
  - Intercepted maps now receive a temporary ID with the `__t_` prefix and are kept in RAM only.
  - An intercepted map is only persisted on the ESP32 when the user explicitly saves it.
  - After renaming an intercepted map, the transient status is correctly cleared; a browser reload no longer reopens the rename dialog.

- **Rename dialog enforced for intercepted maps**
  - The rename dialog is opened automatically as soon as an intercepted map is recognized by the backend and still has no name.
  - The new `"intercepting"` workflow state ensures the dialog blocks until a valid name is entered.

- **Duplicate name validation**
  - Renaming or saving a map now checks whether the name is already in use.
  - Duplicate names trigger a toast error; the dialog cannot be confirmed with a duplicate name.
  - Both backend (`MowerAdapter`) and frontend (`map-workflow`) validate names.

- **Dirty state syncs after browser reload**
  - The unsaved flag is now reliably sourced from the backend (`currentMapUnsaved`) and always applied in the frontend.
  - On reload, an open, unsaved editing state is preserved.

- **Deleting transient maps handled entirely in the backend**
  - `MowerAdapter::deleteMap` removes `__t_` transient maps directly from RAM.
  - If the deleted map was currently loaded, the backend switches to the first persisted map (or resets to an empty map).
  - The frontend only sends the delete request; list and selection updates are driven by the backend's `mapList` response.
  - The incorrect frontend deletion logic has been removed.

- **Transfer current map to backend before switching or saving**
  - Before another map is loaded or saved, the currently edited frontend map is first sent to the backend via `setMap`.
  - This prevents losing intermediate edits when the user switches between maps.

- **Abort draw mode on map change**
  - `onSelectMap`, `onNewMap`, and `onDiscardMap` call `stopDraw()` so an active drawing interaction does not survive across map changes.

- **Map components modularized and confirmation workflows**
  - The map UI has been split into clearer components.
  - Delete, discard, rename, and replace actions now require explicit confirmation.

- **WebSocket chunked transfer more robust**
  - The chunk index is advanced by the number of points actually sent.
  - Other WebSocket messages are correctly serialized and no longer blocked while a chunked map transfer is in progress.
  - A `StateSource` read-lock prevents the backend map from being modified during a chunked transfer.

- **Other backend fixes**
  - The `mowerConsole` handler now tolerates `StatusMessage` entries without a `lines` array.
  - The save button is reliably enabled after map changes.
  - The WebSocket handler previously blocked all messages that contained a map workflow.

### Fixed

- Intercepted map was saved immediately on rename because no map ID existed yet.
- Toast message for duplicate names did not appear because the `Error` import shadowed the global `Error` constructor.
- OK button was disabled for suggested names of new / intercepted maps because the name matched the current name.
- After renaming an intercepted map and reloading the browser, the rename dialog opened again.
- `generateDefaultName()` in `MapManager` was not public and prevented compilation of the transient map features.
- WebSocket handler incorrectly blocked all messages that contained a map workflow.
- Save button was not reliably enabled after map changes.
- Active draw mode continued running when switching or discarding a map.

### Technical Details

- New / extended domain structures:
  - `MapInfo` with `unsaved` and `requiresRename` flags.
  - `TransientMap` in the backend with `requiresRename = true`.
- New WebSocket workflow state `intercepting` in the frontend.
- `MapInfo.requiresRename` is serialized and evaluated over WebSocket.
- `isMapDirty` is updated centrally from the backend sync.
- `StateSource` read-lock during chunked map transfers.

### Known Limitations

- Map changes are only persisted after an explicit save. Discarding or deleting a transient map is irreversible.
